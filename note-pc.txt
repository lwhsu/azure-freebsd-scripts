Partner Center VM Offer Automation
===================================

Prerequisites:
  - curl, jq, diff, mktemp (standard on FreeBSD)
  - az CLI (for SIG image version auto-discovery)
  - config.sh with TENANT_ID, CLIENT_ID, CLIENT_SECRET, SUBSCRIPTION,
    RESOURCE_GROUP, GALLERY_NAME

Scripts overview:
  pc-subr.sh              Shared API library (sourced by other scripts)
  clone-offer.jq          jq filter for resource-tree transformation
  pc-vm-offer-export.sh   List offers and export resource-tree JSON
  pc-vm-offer-clone.sh    Generate new offer JSON from existing offer (local)
  pc-vm-offer-create.sh   Submit configure JSON to create offer in Partner Center
  pc-vm-offer-update.sh   Extract/diff/submit resource updates to existing offer
  pc-vm-offer-review.sh   Human-readable summary of an offer
  pc-vm-offer-compare.sh  Normalized diff of two resource trees
  pc-vm-offer-submit.sh   Submit offer to preview or go live
  pc-vm-offer-status.sh   Check job or submission status

---

Example: Creating FreeBSD 14.4-RELEASE offer
=============================================

1. Export the source offer (14.3) if not already done
-----------------------------------------------------

  NON_INTERACTIVE_OFFER=freebsd-14_3 ./pc-vm-offer-export.sh

  This creates resource-tree_freebsd-14_3_draft.json.
  Skip this step if the file already exists.


2. Generate the new offer JSON locally
--------------------------------------

  From a local file (fully offline if -g is used):

  ./pc-vm-offer-clone.sh -f resource-tree_freebsd-14_3_draft.json -t 14.4

  Or fetch the source directly from the API:

  ./pc-vm-offer-clone.sh -s freebsd-14_3 -t 14.4

  This auto-discovers SIG image versions via az CLI.
  Output: offer-create_freebsd-14_4.json

  Options:
    -T RC1                            SIG image tag (default: RELEASE)
    -g 2026.0301.00                   Use a specific SIG version for all archs
    --arm64-sig-version 2026.0315.00  Override SIG version for arm64 only
    -g none                           Empty vmImageVersions (add images later)
    -o "fifth release"                Override ordinal text (default: auto)
    -O custom-output.json             Override output filename

  The ordinal defaults to the minor version number:
    14.0 -> "first release"
    14.1 -> "second release"
    14.2 -> "third release"
    14.3 -> "fourth release"
    14.4 -> "fifth release"

  Example with explicit SIG versions:

  ./pc-vm-offer-clone.sh -f resource-tree_freebsd-14_3_draft.json \
    -t 14.4 -g 2026.0301.00 --arm64-sig-version 2026.0315.00

  Using RC/BETA images for testing (-T option):

  The -T option changes which SIG image definitions are referenced,
  without changing the offer identity. This is useful for testing an
  offer with RC or BETA images before the final RELEASE images exist.

  ./pc-vm-offer-clone.sh -f resource-tree_freebsd-14_3_draft.json \
    -t 14.4 -T RC1 -g 2026.0301.00

  This creates the offer as freebsd-14_4 (FreeBSD 14.4-RELEASE) but
  the SIG image paths reference FreeBSD-14.4-RC1-amd64-ufs-gen1 etc.
  When RELEASE images are ready, re-clone without -T or update
  the tech config via pc-vm-offer-update.sh.

  termsOfUse license check:

  The clone script automatically fetches the current FreeBSD license
  from https://www.freebsd.org/copyright/freebsd-license/ and compares
  it with the template's termsOfUse text. If only the copyright year
  has changed (e.g., 1992-2025 -> 1992-2026), it auto-updates. If the
  license text itself has changed, it shows a diff and asks whether to
  use the upstream text.


3. Review the generated config
------------------------------

  ./pc-vm-offer-review.sh -f offer-create_freebsd-14_4.json

  Check:
    - Product externalId: freebsd-14_4
    - Plan externalIds: 14_4-release-ufs, 14_4-release-zfs
    - Listing title: FreeBSD 14.4-RELEASE
    - Short description ordinal: "fifth release"
    - Release notes/readme/errata links point to 14.4R
    - SKU IDs: 14_4-release-ufs, 14_4-release-amd64-gen2-ufs, etc.
    - vmImageVersions: correct SIG paths or empty


4. Compare with source to verify only version-specific fields changed
---------------------------------------------------------------------

  ./pc-vm-offer-compare.sh resource-tree_freebsd-14_3_draft.json \
    offer-create_freebsd-14_4.json

  Differences should only be in:
    - Version strings (14.3 -> 14.4, 14_3 -> 14_4)
    - Ordinal text ("fourth release" -> "fifth release")
    - Copyright year in termsOfUse (if auto-updated)
    - SIG image paths/versions
    - No durable IDs in the new file (expected)

  Output is colored and paged through less automatically.


5. (Optional) Edit the generated JSON manually
-----------------------------------------------

  Directly with an editor:

  vi offer-create_freebsd-14_4.json

  Or interactively with vimdiff against the source (edits saved to disk):

  ./pc-vm-offer-compare.sh -v resource-tree_freebsd-14_3_draft.json \
    offer-create_freebsd-14_4.json


6. Create the offer in Partner Center
--------------------------------------

  ./pc-vm-offer-create.sh -f offer-create_freebsd-14_4.json

  This POSTs to /configure, polls for completion, and reports
  the new product durable ID on success.


7. Export the new offer back and verify
---------------------------------------

  NON_INTERACTIVE_OFFER=freebsd-14_4 ./pc-vm-offer-export.sh
  ./pc-vm-offer-review.sh -f resource-tree_freebsd-14_4_draft.json


8. If corrections are needed
----------------------------

  Option A: Submit the full exported resource-tree after editing:

  ./pc-vm-offer-compare.sh -v resource-tree_freebsd-14_3_draft.json \
    resource-tree_freebsd-14_4_draft.json
  ./pc-vm-offer-update.sh -e freebsd-14_4 -r resource-tree_freebsd-14_4_draft.json

  The script automatically:
    - Unwraps the resource-tree envelope
    - Filters out listing-asset (expired SAS URLs), customer-leads, submission
    - Normalizes $schema URLs
    - Fixes stale product references

  The diff shown is normalized (no durable IDs, stable sort) for easy review.

  Option B: Extract a specific resource type, edit, and submit:

  ./pc-vm-offer-update.sh -e freebsd-14_4 -R listing -o listing-edit.json
  vi listing-edit.json

  Preview the diff (no submission):

  ./pc-vm-offer-update.sh -e freebsd-14_4 -r listing-edit.json -d

  Submit the changes (shows normalized diff and asks for confirmation):

  ./pc-vm-offer-update.sh -e freebsd-14_4 -r listing-edit.json

  Other resource types:
    -R plan-listing
    -R virtual-machine-plan-technical-configuration
    -R price-and-availability-offer
    -R price-and-availability-plan


9. Add SIG images later (if clone used -g none)
------------------------------------------------

  ./pc-vm-offer-update.sh -e freebsd-14_4 \
    -R virtual-machine-plan-technical-configuration -o tech-config.json
  vi tech-config.json    # add vmImageVersions entries
  ./pc-vm-offer-update.sh -e freebsd-14_4 -r tech-config.json


10. Submit to preview
---------------------

  ./pc-vm-offer-submit.sh -e freebsd-14_4 -t preview


11. Poll for certification (~24-48h)
-------------------------------------

  ./pc-vm-offer-status.sh -j <JOB_ID> -w

  Or check without waiting:

  ./pc-vm-offer-status.sh -j <JOB_ID>
  ./pc-vm-offer-status.sh -j <JOB_ID> -d    # with full error detail

  Or list submissions for the offer:

  ./pc-vm-offer-status.sh -e freebsd-14_4


12. Go live
-----------

  ./pc-vm-offer-submit.sh -e freebsd-14_4 -t live -s <PREVIEW_SUBMISSION_ID>

  The preview submission ID can be found via:

  ./pc-vm-offer-status.sh -e freebsd-14_4

Quick reference: common operations
===================================

Export an offer:
  NON_INTERACTIVE_OFFER=freebsd-14_3 ./pc-vm-offer-export.sh
  TARGET_TYPE=live NON_INTERACTIVE_OFFER=freebsd-14_3 ./pc-vm-offer-export.sh

Review an offer from API:
  ./pc-vm-offer-review.sh -e freebsd-14_3
  ./pc-vm-offer-review.sh -e freebsd-14_3 -t live

Review an offer from file:
  ./pc-vm-offer-review.sh -f resource-tree_freebsd-14_3_draft.json

Compare two offers (colored diff + pager):
  ./pc-vm-offer-compare.sh resource-tree_freebsd-14_3_draft.json \
    resource-tree_freebsd-15_0_draft.json

Compare and edit interactively in vimdiff:
  ./pc-vm-offer-compare.sh -v resource-tree_freebsd-14_3_draft.json \
    offer-create_freebsd-14_4.json
  (edits are saved back to the original files)

Clone with RC images for testing:
  ./pc-vm-offer-clone.sh -f resource-tree_freebsd-14_3_draft.json \
    -t 14.4 -T RC1 -g 2026.0301.00

Interactive export (pick from list):
  ./pc-vm-offer-export.sh

Set recommended VM sizes:
  ./pc-vm-offer-update.sh -e freebsd-14_4 -r tech-config-recommended-vm-sizes_freebsd-14_4.json


Recommended VM sizes format
============================

Partner Center uses a different format from the standard Azure VM size names
returned by `az vm list-sizes`.

  Azure API name       Partner Center value
  ----------------     --------------------
  Standard_A1_v2       a1-standard-v2
  Standard_D2s_v3      d2s-standard-v3
  Standard_D2s_v5      d2s-standard-v5
  Standard_D2ps_v5     d2ps-standard-v5

Conversion rule: strip "Standard_", move the tier word to the middle,
lowercase everything, underscores -> hyphens.

  Standard_{X}_{vN}  ->  {x}-standard-{vN}

Shell one-liner:
  echo "Standard_D2s_v5" | \
    sed 's/Standard_\(.*\)_\(v[0-9]*\)/\1-standard-\2/' | \
    tr '[:upper:]' '[:lower:]'
  # -> d2s-standard-v5

Note: v6 sizes (e.g. Standard_D2s_v6) are NOT accepted by Partner Center
as of early 2026 even though they are GA in Azure Compute. Use v5.

Current sizes used for FreeBSD 14.4 (both UFS and ZFS plans):
  d2s-standard-v5    (Standard_D2s_v5,  2 vCPU,  8 GiB, Intel x64)
  d4s-standard-v5    (Standard_D4s_v5,  4 vCPU, 16 GiB, Intel x64)
  d8s-standard-v5    (Standard_D8s_v5,  8 vCPU, 32 GiB, Intel x64)
  d2as-standard-v5   (Standard_D2as_v5, 2 vCPU,  8 GiB, AMD x64)
  d2ps-standard-v5   (Standard_D2ps_v5, 2 vCPU,  8 GiB, Ampere arm64)
  d4ps-standard-v5   (Standard_D4ps_v5, 4 vCPU, 16 GiB, Ampere arm64)

Template file: tech-config-recommended-vm-sizes_freebsd-14_4.json
(contains full virtual-machine-plan-technical-configuration resources
 for both plans with recommendedVmSizes populated)
